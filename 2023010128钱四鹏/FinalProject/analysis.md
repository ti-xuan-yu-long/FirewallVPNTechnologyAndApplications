### 3.1 攻击总览

### 3.2 攻击ATTACK-01：内网扫描

#### 3.2.1 攻击详情

| 项目 | 内容 |
|:-----|:-----|
| 攻击名称 | 内网ICMP网段扫描 |
| MITRE ATT&CK编号 | T1046 - Network Service Scanning |
| 攻击来源 | guest (10.30.0.2) |
| 攻击目标 | office网段 10.20.0.1-10 |
| 攻击工具 | bash脚本 + ping命令 |
| 攻击意图 | 探测内网存活主机，寻找攻击目标 |

#### 3.2.2 攻击执行

**攻击命令序列：**
```bash
# 从guest发起ping扫描
for i in {1..10}; do
    sudo ip netns exec guest ping -c 1 -W 1 10.20.0.$i 2>/dev/null && echo "10.20.0.$i is up"
done
```

#### 3.2.3 攻击结果

**执行结果：**
```
所有ping请求均超时，未发现任何存活主机
```

**防火墙日志记录：**
```
kernel: GUEST-TO-OFFICE: IN=veth-fw-guest OUT=veth-fw-office 
SRC=10.30.0.2 DST=10.20.0.1 PROTO=ICMP
kernel: GUEST-TO-OFFICE: IN=veth-fw-guest OUT=veth-fw-office 
SRC=10.30.0.2 DST=10.20.0.2 PROTO=ICMP
kernel: GUEST-TO-OFFICE: IN=veth-fw-guest OUT=veth-fw-office 
SRC=10.30.0.2 DST=10.20.0.3 PROTO=ICMP
...
```

#### 3.2.4 失败原因深度分析

**技术层面：**

| 防御层 | 防御机制 | 说明 |
|:-------|:---------|:-----|
| 第1层 | FORWARD默认DROP | 所有转发流量默认被拒绝 |
| 第2层 | 明确的REJECT规则 | guest→office方向有明确的拒绝规则 |
| 第3层 | LOG记录 | 所有尝试都被记录，形成审计证据 |

**防火墙规则匹配流程：**
```
包进入: guest → fw (ICMP请求 10.30.0.2 → 10.20.0.x)
    ↓
第1步: 匹配ESTABLISHED,RELATED? → 否 (这是新连接)
    ↓
第2步: 匹配office→dmz:8080规则? → 否 (方向不匹配)
    ↓
第3步: 匹配guest→office REJECT规则 → 是 (匹配成功)
    ↓
第4步: 执行LOG规则 → 记录日志
    ↓
第5步: 执行REJECT → 返回ICMP管理禁止
```

**攻击方视角分析：**

攻击者从guest发起扫描，得到的是"icmp-admin-prohibited"响应，这意味着：
1. **明确信号**：网络存在且目标可达，但被管理员策略拒绝
2. **判断依据**：REJECT比DROP提供了更多信息
3. **战术价值**：可以确认"目标存在且受保护"，而非"目标不存在"

### 3.3 攻击ATTACK-02：端口绕过

#### 3.3.1 攻击详情

| 项目 | 内容 |
|:-----|:-----|
| 攻击名称 | 防火墙端口绕过尝试 |
| MITRE ATT&CK编号 | T1090 - Proxy / T1568 - Dynamic Resolution |
| 攻击来源 | guest (10.30.0.2) |
| 攻击目标 | dmz:22 (10.40.0.2) |
| 攻击工具 | curl命令 |
| 攻击意图 | 通过变换源端口绕过防火墙规则 |

#### 3.3.2 攻击执行

**攻击命令序列：**
```bash
# 尝试使用不同源端口
sudo ip netns exec guest curl --local-port 80 --max-time 2 http://10.40.0.2:22/
sudo ip netns exec guest curl --local-port 443 --max-time 2 http://10.40.0.2:22/
sudo ip netns exec guest curl --local-port 1024 --max-time 2 http://10.40.0.2:22/
sudo ip netns exec guest curl --local-port 53 --max-time 2 http://10.40.0.2:22/
```

#### 3.3.3 攻击结果

| 尝试 | 源端口 | 结果 | 日志记录 |
|:----:|:-------|:-----|:---------|
| 1 | 80 | ❌ 失败 | GUEST-TO-DMZ |
| 2 | 443 | ❌ 失败 | GUEST-TO-DMZ |
| 3 | 1024 | ❌ 失败 | GUEST-TO-DMZ |
| 4 | 53 | ❌ 失败 | GUEST-TO-DMZ |

#### 3.3.4 失败原因深度分析

**技术原理：**

iptables规则匹配是基于以下五元组的组合：
```
匹配条件 = {协议, 源IP, 源端口, 目标IP, 目标端口}
```

本场景中，防火墙规则为：
```
-i veth-fw-guest -o veth-fw-dmz -p tcp --dport 22 -j REJECT
```

| 匹配字段 | 规则要求 | 攻击实际 | 是否匹配 |
|:---------|:---------|:---------|:---------|
| 输入接口 | veth-fw-guest | veth-fw-guest | ✅ |
| 输出接口 | veth-fw-dmz | veth-fw-dmz | ✅ |
| 协议 | TCP | TCP | ✅ |
| 目标端口 | 22 | 22 | ✅ |
| 源端口 | 不限制 | 80/443/1024/53 | ✅ (不限制) |

**关键结论：** 源端口不在防火墙的匹配条件中，因此变换源端口无法绕过规则。

**攻击方视角分析：**

这个攻击尝试揭示了几个重要事实：
1. 防火墙规则是"基于策略"而非"基于会话"的
2. 源端口的随机化不能绕过基于方向的访问控制
3. 攻击者需要寻找其他绕过途径（如协议层面、应用层面）

### 3.4 攻击ATTACK-03：VPN伪造

#### 3.4.1 攻击详情

| 项目 | 内容 |
|:-----|:-----|
| 攻击名称 | VPN源地址伪造攻击 |
| MITRE ATT&CK编号 | T1206 - Trusted Relationship |
| 攻击来源 | guest (10.30.0.2) |
| 攻击目标 | dmz (10.40.0.2) |
| 攻击工具 | ping -I 参数 |
| 攻击意图 | 伪造VPN地址绕过访问控制 |

#### 3.4.2 攻击执行

**攻击命令：**
```bash
sudo ip netns exec guest ping -c 1 -I 10.10.10.2 10.40.0.2
```

**执行结果：**
```
ping: local error: Cannot assign requested address
```

#### 3.4.3 失败原因深度分析

**三层防御机制：**

```
┌─────────────────────────────────────────────────────────────────────┐
│                     三层防御剖析                                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  第1层: 网络层地址验证                                              │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  10.10.10.2 地址不属于 guest 的任何接口                     │   │
│  │  → Linux内核拒绝绑定非本地IP                                │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                              ↓ 失败                                │
│                                                                     │
│  第2层: rp_filter 反向路径过滤                                     │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  即使攻击成功伪造，rp_filter会检查源地址的路由路径          │   │
│  │  → 源地址10.10.10.2的路由出口不是guest接口                 │   │
│  │  → 内核丢弃该包                                            │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
│  第3层: WireGuard加密隧道保护                                      │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │  WireGuard需要预共享公钥进行加密/解密                       │   │
│  │  → 未经加密的包无法进入wg0接口                             │   │
│  │  → 即使进入，公钥验证失败也会被丢弃                        │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**rp_filter工作原理：**

```bash
# 查看rp_filter状态
cat /proc/sys/net/ipv4/conf/all/rp_filter
# 输出: 1 (启用)

# rp_filter检查逻辑
# 包到达: SRC=10.10.10.2, IN=veth-fw-guest
# 检查: 反向路由查询 10.10.10.2 应该从哪个接口出去?
# 结果: 从wg0接口出去 (因为10.10.10.0/24在wg0上)
# 结论: 入接口(veth-fw-guest) ≠ 出接口(wg0) → 丢弃!
```

**攻击方视角分析：**

这个攻击揭示了：
1. VPN地址不是简单的"通行证"，需要完整的加密认证
2. 即使绕过网络层，rp_filter也会提供保护
3. WireGuard的加密隧道提供了强有力的边界保护
4. 多层防御使单一攻击手段几乎不可能成功

### 3.5 REJECT vs DROP 攻防分析

| 对比维度 | REJECT | DROP | 攻防意义 |
|:---------|:-------|:-----|:---------|
| 响应时间 | 立即(ms级) | 超时(s级) | REJECT让攻击者扫描更快 |
| 返回信息 | ICMP不可达/RST | 无 | REJECT暴露目标存在 |
| 目标判断 | ✅ 确定存在 | ❌ 无法判断 | REJECT降低攻击成本 |
| 扫描效率 | 高 | 低 | REJECT提高攻击效率 |
| 隐蔽性 | 低 | 高 | DROP更利于防御 |
| 故障定位 | 易 | 难 | REJECT便于运维 |
| 推荐场景 | 内网/调试 | 外网/生产 | - |

**攻防建议：**

| 场景 | 推荐策略 | 理由 |
|:-----|:---------|:-----|
| 外网接口 | DROP | 提高扫描成本，不暴露目标信息 |
| 内网接口 | REJECT | 便于故障排查，快速定位问题 |
| DMZ接口 | REJECT(调试)/DROP(生产) | 根据环境选择 |
| VPN接口 | REJECT | 明确告知VPN用户权限不足 |


## 四、防御方分析

### 4.1 日志分析与攻击识别

#### 4.1.1 日志字段安全价值分析

**一条完整的拒绝日志示例：**
```
kernel: [12345.678901] GUEST-TO-OFFICE: IN=veth-fw-guest OUT=veth-fw-office 
SRC=10.30.0.2 DST=10.20.0.5 PROTO=ICMP
```

| 字段 | 示例值 | 安全含义 | 威胁情报价值 |
|:-----|:-------|:---------|:-------------|
| IN | veth-fw-guest | 攻击来源区域 | 识别攻击入口点 |
| OUT | veth-fw-office | 攻击目标区域 | 识别攻击方向 |
| SRC | 10.30.0.2 | 攻击者IP | 定位攻击源 |
| DST | 10.20.0.5 | 受害者IP | 定位攻击目标 |
| PROTO | ICMP | 攻击类型 | 识别扫描行为 |
| DPT | 22 | 目标端口 | 识别特定服务攻击 |
| SPT | 54321 | 源端口 | 辅助追踪连接 |

#### 4.1.2 攻击行为识别矩阵

| 日志特征 | 可能攻击类型 | 威胁级别 | 响应建议 |
|:---------|:-------------|:---------|:---------|
| 大量ICMP到不同目标 | 网段扫描 | 低 | 记录观察 |
| 大量TCP SYN到22端口 | SSH暴力破解 | 高 | 立即封禁 |
| 单一源访问多个端口 | 端口扫描 | 中 | 速率限制 |
| guest→office任何流量 | 内部横向移动 | 高 | 调查来源 |
| VPN→dmz:22 | 权限滥用 | 中 | 审查用户 |

#### 4.1.3 本次演练攻击检测结果

| 攻击 | 检测方式 | 检测时间 | 响应动作 |
|:-----|:---------|:---------|:---------|
| 网段扫描 | LOG规则匹配 | 实时 | 记录+速率限制 |
| 端口绕过 | LOG规则匹配 | 实时 | 记录+拒绝 |
| VPN伪造 | 系统层阻止 | 网络层 | 自动丢弃 |

### 4.3 防御有效性评估

| 评估维度 | 评分 | 说明 |
|:---------|:----:|:-----|
| 边界防御 | ⭐⭐⭐⭐⭐ | FORWARD DROP + 状态检测，有效阻止所有未授权访问 |
| 区域隔离 | ⭐⭐⭐⭐⭐ | guest完全隔离，office受限访问，DMZ受控暴露 |
| 访问控制精度 | ⭐⭐⭐⭐⭐ | 基于IP+端口+协议的精确控制 |
| 日志审计 | ⭐⭐⭐⭐ | 完整记录所有拒绝事件，速率限制保护系统 |
| VPN安全 | ⭐⭐⭐⭐⭐ | WireGuard加密+AllowedIPs控制+防火墙三重保护 |
| 综合防御 | ⭐⭐⭐⭐⭐ | 纵深防御体系，无单点故障 |


## 五、红蓝对抗总结

### 5.1 攻击方总结

**成功的攻击尝试：**
- 无明显成功的攻击突破
- 所有攻击均被防火墙有效拦截

**失败的攻击尝试：**
| 攻击 | 失败原因 | 经验教训 |
|:-----|:---------|:---------|
| 网段扫描 | FORWARD DROP + 明确REJECT | 最小权限原则有效防御扫描 |
| 端口绕过 | 基于策略的防火墙 | 变换源端口无法绕过方向控制 |
| VPN伪造 | 三层防御机制 | 多层防御使单一攻击失效 |

**攻击方战术建议：**

1. **信息收集优先**：先通过公开信息了解目标网络架构
2. **寻找薄弱环节**：关注应用层漏洞而非网络层
3. **社交工程**：人是最薄弱的环节
4. **0day利用**：网络层防御较难，应用层仍有突破可能

### 5.2 防御方总结

**有效的防御措施：**
| 措施 | 效果 | 优化建议 |
|:-----|:-----|:---------|
| FORWARD DROP | ✅ 有效阻止所有未授权流量 | 保持默认策略 |
| 状态检测 | ✅ 保证正常业务通信 | 确保规则顺序正确 |
| 区域隔离 | ✅ 完全隔离guest区域 | 增加应用层过滤 |
| LOG审计 | ✅ 完整记录攻击行为 | 增加实时告警 |
| VPN控制 | ✅ 精确控制VPN访问 | 增加会话超时 |
| 速率限制 | ✅ 防止日志洪水 | 调整适合阈值 |

**防御改进建议：**

1. **增加IDS/IPS**：在网络边界部署入侵检测/防御系统
2. **添加WAF**：为DMZ Web服务增加应用层防火墙
3. **实施零信任**：内部网络增加微隔离和持续认证
4. **自动化响应**：结合SIEM实现自动封禁
5. **定期演练**：定期进行红蓝对抗演练

### 5.3 攻防对抗时间线

**攻击行为时间轴记录表**
| 时间轴 | 攻击方动作             | 防御方响应                     |
| ------ | ---------------------- | ------------------------------ |

### 5.4 关键发现与结论

| 发现编号 | 关键发现 | 安全意义 |
|:--------:|:---------|:---------|
| F-01 | REJECT响应暴露目标存在 | 生产环境外网应使用DROP |
| F-02 | 源端口变换无法绕过防火墙 | 基于策略的防火墙是有效的 |
| F-03 | 三层防御保护VPN | 单一防御不足以保护VPN |
| F-04 | LOG规则完整记录攻击 | 日志审计是安全运营的基础 |
| F-05 | 速率限制保护系统 | 防止日志洪水和资源耗尽 |
| F-06 | 规则顺序至关重要 | 顺序错误导致防御失效 |
