# 攻防演练分析报告

## 一、引言

本次攻防演练在构建完成的企业级多区域隔离网络基础上，从攻击方视角尝试突破边界防护，从防御方视角分析安全策略的有效性，并通过边界加固和流量追踪进一步完善安全体系。演练覆盖了从信息收集、漏洞探测到权限维持的完整攻击链，验证了"分区隔离 + 精细管控 + 审计追溯"三位一体防御架构的实际效果。

## 二、攻击方视角：从访客网发起的渗透尝试

### 2.1 攻击1：ICMP网段扫描——信息收集的失效

**攻击过程**：从 guest 命名空间向 office 网段发起 ICMP 扫描，尝试探测存活主机。

```bash
for i in {1..10}; do
  sudo ip netns exec guest ping -c 1 -W 1 10.20.0.$i 2>/dev/null && echo "10.20.0.$i is up"
done
```

**结果分析**：扫描结果显示，除网关 `10.20.0.1`（防火墙接口）可达外，其余主机全部返回 `Destination Host Unreachable`。表面上看攻击者获得了一条"有用"信息——网关存活，但实际上这正是防火墙 REJECT 策略的特征：返回明确的拒绝报文让攻击者迅速获得反馈。

**深层思考**：这里暴露出 REJECT 策略的一个固有矛盾——在便于运维调试的同时，也向攻击者暴露了防火墙的存在。如果改用 DROP 策略，攻击者将无法区分"主机不存在""端口未开放""防火墙拦截"，探测成本将指数级上升。这也解释了为什么生产环境的边界防护通常采用 DROP 而非 REJECT：宁可牺牲一些调试便利，也要增加攻击者的信息收集难度。

**攻击本质**：防火墙的策略组合是"默认 DROP + 特定 REJECT"，对于 guest→office 这种明确禁止的访问，规则在匹配后执行 REJECT。网关可达是因为回包由防火墙自身生成，并没有触及办公网内部。攻击者无法获取内网存活主机的任何信息，内网资产得以隐藏，攻击链在第一步就被切断。

### 2.2 攻击2：源端口篡改——绕过策略的无效尝试

**攻击过程**：攻击者尝试修改本地源端口，试图绕过访问控制。

```bash
sudo ip netns exec guest curl --local-port 80 --max-time 2 http://10.40.0.2:22/
sudo ip netns exec guest curl --local-port 443 --max-time 2 http://10.40.0.2:22/
```

**结果分析**：无论源端口改为 80 还是 443，均返回 `Connection refused`。防火墙规则匹配的是五元组中的入接口（`veth-fw-guest`）、目标 IP（`10.40.0.2`）和目标端口（22），源端口不在匹配条件内。这种"白名单"策略只关心访问者来自哪个区域、要去哪里、访问什么服务，而不关心访问者从哪个端口出发。

**深层思考**：这个攻击的失败揭示了防火墙设计的核心哲学——基于"你是谁"和"你要去哪"而非"你怎么来的"。源端口对攻击者来说是可控变量，但对防火墙来说是不可信因素。真正的安全策略应该建立在稳定的标识上（如区域、身份），而非攻击者可随意篡改的临时属性。这也让我联想到现代零信任架构的核心思想：永远基于可信身份而非网络位置来做决策。

### 2.3 攻击3：IP伪造——多层防护的协同阻断

**攻击过程**：攻击者尝试伪造 VPN 客户端源地址 `10.10.10.2`，绕过区域限制直接访问内网。

```bash
sudo ip netns exec guest hping3 -S -a 10.10.10.2 -p 22 10.40.0.2
```

**结果分析**：158 个伪造 SYN 包全部被丢弃，0 回复。这背后是三重防护机制的协同作用：

1. **接口层隔离**：伪造包从 `veth-fw-guest` 进入，而 VPN 允许规则要求入接口为 `wg0`，接口不匹配直接导致规则无法命中。
2. **反向路径过滤（rp_filter）**：Linux 内核会校验源地址的可达性——`10.10.10.2` 不在 guest 网段路由表中，包被直接丢弃。
3. **加密认证**：即使前两层被绕过，WireGuard 的加密和公钥认证机制也会拒绝非授权数据。

**深层思考**：这个攻击的失败最有教育意义——它展示了纵深防御的威力。没有任何一层防御是完美的，但当它们叠加在一起时，攻击者需要同时突破多重障碍。更关键的是，这三层防御工作在不同的抽象层次：接口隔离在网络层、rp_filter 在路由层、WireGuard 在传输加密层。攻击者需要掌握不同层次的知识并找到各自的漏洞，单一攻击手法完全无法奏效。

## 三、防御方视角：从日志到规则的审计链条

### 3.1 审计环境的技术权衡

本实验运行于 WSL 环境，`iptables LOG` 无法被 `journalctl` 或 `dmesg` 捕获。这反映了技术选型中的现实权衡：WSL 提供了便捷的 Linux 开发环境，但在网络功能上与原生 Linux 内核存在差异。面对这一限制，采用规则计数器（`pkts` 列）作为等效的审计证据，每次规则匹配都会使计数器增加，足以证明 LOG 规则已被触发。

**深层思考**：这个限制实际上揭示了一个更本质的问题——安全审计的可靠性取决于日志收集链路的完整性。在真实生产环境中，日志可能经过 syslog、rsyslog、远程日志服务器、SIEM 系统等多个环节，任何一环出问题都可能导致审计盲区。WSL 环境的问题放大了这个风险，也提醒我们在设计企业安全架构时必须确保日志链路的冗余和可验证性。

### 3.2 规则防御效果评估

从计数器输出可以清晰看到：

- 规则 3（`GUEST-TO-OFFICE` LOG）命中 17 次，规则 7（guest→office REJECT）同样命中 17 次，一一对应，说明每次违规访问都被完整记录并拦截。
- 规则 2（`GUEST-TO-DMZ`）命中 5 次，规则 8 同样命中 5 次，模式一致。
- VPN 规则：规则 18 放行 office 访问，规则 19 放行 dmz:8080，规则 20-21 记录并拒绝 dmz:22。

**关键洞察**：LOG 与 REJECT 的计数器完全匹配，证明每条拒绝都有对应的审计记录，满足了安全合规中的"可追溯"要求。同时也验证了规则顺序的正确性：如果 LOG 放在 REJECT 之后，计数器将不会增加。

## 四、边界加固：出站控制的实战意义

### 4.1 隐患识别：被忽视的出站风险

在演练中识别出一个容易被忽视的安全盲点：办公网可无条件访问互联网（`-o veth-fw-inet -j ACCEPT`）。企业安全建设往往集中于入站防护，但出站流量的风险同样巨大：员工可能下载恶意软件、内部主机可能被植入后门主动外联、敏感数据可能被违规外传。

**创新发现**：更隐蔽的风险在于，出站无限制意味着防火墙的 FORWARD 链对办公网出站流量不做任何状态检测和应用层识别。攻击者如果控制了办公网内的一台主机，可以自由建立反向隧道、C2 通信或数据外传，而防火墙层面完全无法感知。这是"无限制出站"最危险的衍生风险。

### 4.2 加固方案

```bash
# 删除原有全量放行规则
sudo ip netns exec fw iptables -D FORWARD 11

# 仅放行业务必需端口
sudo ip netns exec fw iptables -A FORWARD \
  -s 10.20.0.0/24 -o veth-fw-inet \
  -p tcp -m multiport --dports 80,443,53 \
  -m conntrack --ctstate NEW -j ACCEPT

sudo ip netns exec fw iptables -A FORWARD \
  -s 10.20.0.0/24 -o veth-fw-inet \
  -p udp -m multiport --dports 53,123 \
  -m conntrack --ctstate NEW -j ACCEPT
```

**设计逻辑**：仅放行 Web 浏览（80/443）、DNS（53）、时间同步（123）等业务必需端口，其他端口全部被默认 DROP 拦截。这遵循了最小权限原则——只给工作必需的权限，不给额外权限。

**深层思考**：这种加固方案虽然有效，但仍有局限性。它基于端口进行白名单控制，无法识别应用层行为（如 HTTP 请求是否包含恶意 payload）。真正的企业安全需要在此之上叠加 Web 过滤、DLP 等应用层安全能力。但作为基础边界防护，端口白名单已经能拦截大部分非授权外联，是成本最低、效果最明显的安全措施。

## 五、VPN 流量追踪：加密隧道的可视化管理

### 5.1 全链路抓包分析

| 阶段 | 观测位置 | 源地址 | 目的地址 | 协议 | 备注 |
|:---|:---|:---|:---|:---|:---|
| 1 | remote wg0 | 10.10.10.2:47892 | 10.40.0.2:8080 | TCP | VPN 封装前的原始业务包 |
| 2 | fw wg0 | 10.10.10.2:47892 | 10.40.0.2:8080 | TCP | 防火墙解封装后的内网包 |
| 3 | fw veth-fw-dmz | 10.10.10.2:47892 | 10.40.0.2:8080 | TCP | 转发至 DMZ 服务器的业务包 |
| 4 | conntrack | 10.10.10.2:45825 | 10.10.10.1:51820 | UDP | WireGuard 隧道保活会话 |

### 5.2 关键发现

1. **IP 层透明性**：WireGuard 在 IP 层（网络层）工作，封装和解封装不改变原始数据包的 IP 头和传输层头，只在外部增加 UDP 封装。这保证了应用层协议（HTTP）在通过隧道时完全不受影响。

2. **连接状态分离**：conntrack 记录的是 WireGuard 的 UDP 控制会话（用于 keepalive），而非业务 TCP 会话。这体现了 VPN 的数据平面和控制平面分离设计。

3. **动态地址转换**：remote 的源端口从 45825（WireGuard 通信端口）到 47892（业务连接源端口）的变化，展示了 VPN 隧道的多路复用能力——一个隧道承载多条业务连接。

**深层思考**：从安全监控角度看，这种设计意味着管理员需要同时关注两个层面的流量：隧道层（是否有异常的大流量数据传输）和业务层（隧道内传输的业务是否符合安全策略）。在真实企业环境中，这通常需要部署 NDR（网络检测与响应）等高级监控工具，在不解密流量的情况下通过元数据分析发现异常。

## 六、攻击者行为分析与 REJECT/DROP 的战术意义

### 6.1 REJECT 与 DROP 的信息暴露差异

| 行为 | REJECT | DROP |
|:---|:---|:---|
| 响应行为 | 返回 ICMP 不可达或 TCP RST | 无响应 |
| 客户端感知 | 立即失败（`Connection refused`） | 超时等待 |
| 攻击者可判断 | 目标在线，端口被防火墙拒绝 | 无法区分目标不存在/端口关闭/防火墙拦截 |
| 扫描速度 | 快（立即返回，适合快速探测） | 慢（需等待超时） |
| 信息暴露 | 暴露防火墙存在 | 隐藏防火墙存在 |

### 6.2 战术建议

- **内部区域之间**（如 guest→office）：使用 REJECT，便于运维人员快速发现违规访问并及时处置。
- **外部到内部**（如 internet→内网）：使用 DROP 或默认策略，增加攻击者扫描成本和信息收集难度。
- **调试环境**：优先使用 REJECT，快速定位规则问题。
- **生产环境**：对外边界使用 DROP，对内边界可保留 REJECT。

**深层思考**：REJECT 与 DROP 的选择本质上是对"安全"和"可运维"的权衡。追求极致安全会选择 DROP，让攻击者在黑暗中摸索；追求高效运维会选择 REJECT，让问题快速暴露。企业应根据自身的威胁模型和运营能力做出选择——安全是投入产出比的平衡，而非某一技术的极致应用。

## 七、创新性发现与深度思考

### 7.1 WSL 环境下的审计策略调整

面对 WSL 环境无法输出内核日志的限制，没有停留在"无法完成审计任务"的层面，而是主动寻找替代方案——利用 iptables 计数器作为等效证据。这体现了一种重要的安全思维：审计的核心是"可追溯"，而非特定的技术实现方式。当原有技术手段受限于环境时，应以目标为导向寻找等价替代。

### 7.2 出站控制的隐蔽风险

办公网无限制出站的风险远不止于"员工访问恶意网站"这一表面问题。更深层的风险在于：如果内部主机被攻破，攻击者可以自由建立 C2 通道、加密挖矿、数据外传，而防火墙对出站流量的"全放行"策略相当于为攻击者提供了免费的通信管道。这种风险的隐蔽性在于——它不会产生任何告警日志，因为流量是"合法"的。

### 7.3 多层防御的协同价值

IP 伪造攻击的失败展示了纵深防御的核心价值：单一防御可能被绕过，但多层防御叠加后，攻击面大幅缩小，攻击成本指数级上升。特别是接口隔离、反向路径过滤、加密认证这三层工作在不同层面（链路层/路由层/应用加密层）的防护机制，让单一攻击手法无法通杀。

### 7.4 流量抓包体现的加密隧道安全管理需求

从抓包分析可以看出，VPN 隧道的管理需要同时关注两个层面的流量特征：隧道层的 UDP 会话（保活、握手）和业务层的 TCP 会话（实际访问内容）。在不解密流量的前提下，通过元数据分析（流量模式、连接频率、数据量）仍可发现异常行为，这为加密流量的安全监控提供了可行的技术路径。

## 八、总结

本次攻防演练从攻击方（信息收集→权限提升→横向移动）和防御方（规则审计→边界加固→流量追溯）两个视角完整验证了企业安全架构的防御深度。核心收获包括：

1. **分区隔离 + 精细管控**是安全防御的基石，网络命名空间为不同安全等级的区域提供了物理级的隔离边界。

2. **审计与拦截必须协同**，LOG 规则必须配置在 REJECT 之前，且可通过计数器验证有效性。

3. **出站控制是容易被忽视的防护盲点**，需要与入站防护同等重视，遵循最小权限原则进行白名单管控。

4. **VPN 的精细路由设计**（Split Tunnel）是平衡安全与性能的关键，通过 AllowedIPs 实现最小权限的远程接入。

5. **REJECT/DROP 的选择是安全与运维的平衡**，应根据场景灵活选择，内部调试用 REJECT，生产外边界用 DROP。

6. **多层防御的价值在于攻击成本的累积**，单层防御可能被绕过，但层层叠加后攻击难度指数级上升。

安全不是单一技术的成败，而是分层防御、精细管控、持续审计、定期验证的体系化能力。本次攻防演练验证了这一体系的防御效果，也为后续安全建设提供了实践参考。
