## 日志分析报告

### 1、从日志中能获取哪些安全信息

答：通过 iptables LOG 规则可以获取攻击来源、目标、类型和频率等关键信息。SRC 和 IN 字段标识攻击来源区域，如 IN=veth-fw-guest 表示访客网攻击。DST 和 DPT 字段揭示攻击目标，如 DST=10.40.0.2 DPT=22 表示针对 DMZ 的 SSH 攻击。PROTO 字段区分协议类型，结合端口判断是 Web 攻击、暴力破解还是扫描行为。统计单位时间内日志数量可识别 DDoS 或暴力破解等高频攻击。对比 LOG 与 REJECT 计数器可验证审计机制完整性。

### 2、LOG 规则为什么要放在 REJECT 之前

答：iptables 按顺序匹配规则，REJECT 会终止匹配流程，导致后续 LOG 无法触发。LOG 只记录日志并返回继续匹配，数据包最终到达 REJECT 被拦截。若 REJECT 在前，被拒绝的访问将无记录，形成审计盲区。正确顺序必须是 LOG 在前、REJECT 在后，确保每次拦截都有据可查。

### 3、速率限制如何防止日志洪水攻击

答：limit 5/min burst 10 将日志频率控制在每分钟 5 条，允许突发 10 条。这防止攻击者通过高频请求产生海量日志占满磁盘，同时适应正常业务波动。既保护系统资源，又过滤冗余信息，保留有价值的攻击样本便于分析。

### 4、不同 log-prefix 的作用

答：不同前缀用于分类安全事件：GUEST-TO-OFFICE 检测访客越权，GUEST-TO-DMZ 监控访客攻击服务器，VPN-TO-DMZ-SSH 发现 VPN 用户违规操作，INET-TO-OFFICE 识别外部入侵，VPN-DENY 捕获其他 VPN 异常。前缀实现快速分类、便捷统计、分级告警和攻击溯源，提升安全运营效率。

## 七、第五部分：攻防演练

### 攻击演练
1. **扫描攻击**：guest ping扫描office网段，全部被防火墙REJECT
2. **绕过攻击**：尝试改变源端口访问dmz:22，失败（iptables基于目的端口过滤）
3. **伪造攻击**：无法伪造VPN地址（WireGuard使用公钥加密认证）

### 攻击方任务分析

**攻击1** 扫描office网段：从 guest 网段执行 ping 扫描 10.20.0.0/24，探测办公网存活主机。结果所有探测包被防火墙 REJECT 拦截，返回 ICMP 端口不可达错误。失败原因是防火墙明确禁止 guest 访问 office 网段，任何跨区访问都会被规则6拒绝。

**攻击2** 尝试绕过防火墙访问dmz:22：使用 curl 的 --local-port 参数将源端口改为 80 和 443，尝试绕过目的端口过滤访问 dmz 的 22 端口。结果两次尝试均失败，返回连接被拒绝错误。失败原因是 iptables 基于目的端口过滤，与源端口无关，改变源端口无法改变数据包的目的端口 22，因此仍然匹配拒绝规则。

**攻击3** 尝试伪造VPN流量：思考能否伪造源地址为 10.10.10.2 的包访问内网。答案是不能。WireGuard 使用公钥加密进行身份认证，不仅仅是检查源 IP 地址。即使攻击者伪造了 10.10.10.2 的源地址，但由于没有正确的私钥，无法完成 WireGuard 的握手过程，隧道无法建立，因此伪造源地址的攻击不会成功。

REJECT 和 DROP 的区别：REJECT 返回 ICMP 错误响应，客户端立即知道连接被拒绝；DROP 静默丢弃不响应，客户端需要等待超时。攻击者可以从 REJECT 的响应确认目标存在但端口关闭，从 DROP 的超时无法判断目标是否存在。对于已暴露的 DMZ 服务使用 REJECT 更合适，对于隐藏的内网使用 DROP 更安全。

### 防御分析
- 从日志字段 `IN=veth-fw-guest` 可判断攻击来源
- 规则计数器显示guest→office规则有匹配，说明有扫描行为
- REJECT比DROP更安全（攻击者无法通过超时判断目标是否存在）

### 防御方任务分析

**从日志中识别攻击**：从日志的 IN 字段可以判断攻击来源，IN=veth-fw-guest 表示来自 guest 网段。SRC 字段进一步确认具体源 IP 地址。如果日志中 IN=veth-fw-guest OUT=veth-fw-office，说明数据包试图从 guest 网段访问 office 网段，是一次跨区域的越权访问尝试。看到大量相同来源的日志应引起警惕，这表明可能存在自动化扫描或暴力破解攻击，正常用户访问频率较低且分散，而攻击工具通常高频规律发送请求。

**分析规则的防御效果**：规则6拦截了 guest 访问 office，具体为 -i veth-fw-guest -o veth-fw-office -j REJECT。如果 guest 到 office 的规则计数很高，说明 guest 网段存在大量尝试访问 office 的行为，可能是恶意用户越权、设备被入侵作为跳板、或网络中存在扫描探测。REJECT 和 DROP 在安全性上的区别是 REJECT 返回 ICMP 错误告知对方连接被拒绝，客户端快速知道失败；DROP 静默丢弃不响应，客户端等待超时。REJECT 的优点是体验好，缺点是攻击者能确认目标存在；DROP 的优点是隐藏目标，缺点是体验差。

### 边界测试改进
- 添加了连接数限制（connlimit），防止单IP对dmz:8080的DDoS攻击

### 边界测试改进详细分析

选择的问题：dmz:8080 对外开放。

风险分析：DMZ 的 8080 端口通过 DNAT 映射到公网，面临 DDoS 攻击和 Web 漏洞利用风险。攻击者可能发起 SYN Flood 耗尽连接资源，或利用 Web 应用漏洞入侵服务器。一旦 DMZ 被攻破，攻击者可能进一步渗透内网，造成更大范围的安全事故。

改进方案：使用 connlimit 限制单 IP 连接数。具体规则为在 FORWARD 链插入第一条规则，匹配 TCP SYN 包且目的端口为 8080、目的地址为 10.40.0.2，使用 connlimit 模块限制同一 IP 的连接数超过 10 个时拒绝，返回 TCP RST 包。

测试效果：当正常访问时，连接数在限制范围内，访问正常。当使用工具模拟大量连接时，超过 10 个连接后新连接被拒绝，返回连接重置错误。通过 iptables 计数器可以看到 connlimit 规则的匹配次数，证明防护生效。

## 八、故障排查

### 场景1：DNAT配置了但外网无法访问
- **原因**：缺少对应的FORWARD规则
- **排查**：使用conntrack发现无连接记录
- **修复**：添加FORWARD规则放行DNAT后的流量

### 场景2：VPN隧道握手正常但业务访问失败
- **原因**：VPN接口名不匹配（wg0 vs vpn-fw）
- **排查**：检查wg show接口名和FORWARD规则中的接口名
- **修复**：统一使用实际的接口名vpn-fw

### 场景3：状态检测规则缺失
- **原因**：删除了ESTABLISHED,RELATED规则
- **排查**：抓包发现SYN-ACK被拦截
- **修复**：恢复状态检测规则

## 九、遇到的问题和解决方法

1. **问题**：INPUT链默认DROP导致ping不通
   - **解决**：添加允许内部接口的INPUT规则

2. **问题**：VPN接口名不匹配（wg0 vs vpn-fw）
   - **解决**：使用实际的接口名vpn-fw配置FORWARD规则

3. **问题**：curl访问了错误的地址
   - **解决**：确认internet主机的IP地址为203.0.113.10

4. **问题**：iptables LOG在Kali中无法通过journalctl查看
   - **解决**：使用FORWARD链计数器作为日志审计的证据

## 十、总结与思考

通过本次企业级网络安全架构搭建与攻防演练实验，我对企业网络安全有了更深入的理解和认识。
**1、企业网络安全架构的整体理解**
企业网络安全架构的核心是分层防御和区域隔离。本次实验将网络划分为办公区、访客区、DMZ 区和外网四个区域，每个区域有不同的安全级别和访问需求。防火墙作为边界网关，通过 FORWARD 链控制跨区域流量，通过 NAT 实现内外网地址转换，通过 VPN 提供安全的远程接入。这种分层架构的优势在于：即使某一层被突破，攻击者仍需面对其他层的防御，大大增加了攻击难度。
**2、最小权限原则的重要性**
最小权限原则是本次实验贯穿始终的设计理念。在防火墙规则配置中，我们只开放了必要的端口和服务，其余全部默认拒绝。例如，办公网只能访问 DMZ 的 8080 端口，不能访问 22 端口；访客网只能访问外网，不能访问任何内网资源。这种设计有效限制了攻击面，即使某个区域被入侵，攻击者也无法轻易横向移动到其他区域。最小权限原则不仅适用于网络访问控制，也适用于系统权限分配、应用权限设计等各个方面，是安全建设的基石。
**3、VPN 在企业远程接入中的应用**
WireGuard VPN 的配置让我深刻理解了安全远程接入的实现原理。通过公钥加密建立隧道，确保数据传输的机密性和完整性；通过 AllowedIPs 精细控制路由，确保只有访问内网的流量走 VPN，不影响正常上网；通过防火墙规则进一步限制 VPN 用户的访问范围，实现二次授权。这种设计既保证了远程办公的便利性，又确保了内网资源的安全性。在实际企业中，VPN 是疫情期间远程办公的重要基础设施，其安全性直接关系到企业核心数据的安全。
**4、日志审计的必要性**
日志审计是安全运营的眼睛。虽然本次实验中 Kali 的 journalctl 无法直接显示 iptables LOG，但通过 FORWARD 链计数器，我们仍然验证了日志规则的有效性。在实际生产环境中，日志用于攻击溯源、行为分析、合规审计和威胁情报。没有日志，安全事件发生后无法追溯原因，无法定位攻击者，无法评估损失。因此，完善的日志收集、存储和分析体系是企业安全建设的必备组件。
**5、实验中的收获与反思**
本次实验让我将理论知识转化为实践能力。从 namespace 和 veth 的创建，到 iptables 规则的编写，再到 WireGuard 隧道的建立，每一步都需要精确配置和反复验证。遇到问题时，如 INPUT 链默认 DROP 导致 ping 不通、VPN 接口名不匹配导致规则失效，都需要耐心排查和逻辑分析。这些经历培养了我的故障排查能力和问题解决思维，为今后从事网络安全工作打下了坚实基础。
