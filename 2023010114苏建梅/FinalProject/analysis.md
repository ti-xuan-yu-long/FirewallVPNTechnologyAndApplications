
# 攻防演练分析报告

## 一、概述

本次攻防演练基于已搭建的企业边界网络安全环境，从攻击者和防御者两个视角展开。攻击方从guest区域发起三类典型攻击，防御方通过日志分析、规则计数器识别攻击行为，并对边界安全进行改进。演练旨在检验防火墙策略的有效性、日志审计的完整性以及安全防护的纵深程度。


## 二、攻击方演练

### 2.1 攻击背景

攻击起点为guest区域（10.30.0.2），该区域代表访客网络，信任级别最低。攻击者可能通过访客设备接入企业网络，尝试探测内网结构、横向移动到更高价值区域（办公网、DMZ），或绕过访问控制获取未授权服务。

### 2.2 攻击1：扫描office网段

**攻击目标：** 探测office区域（10.20.0.0/24）存活主机，为后续横向移动收集信息。

**攻击命令：**
```bash
for i in {1..10}; do
  sudo ip netns exec guest ping -c 1 -W 1 10.20.0.$i 2>/dev/null && echo "10.20.0.$i is up"
done
```

**攻击结果：**

```
PING 10.20.0.1 (10.20.0.1) 56(84) bytes of data.
64 bytes from 10.20.0.1: icmp_seq=1 ttl=64 time=0.079 ms
10.20.0.1 is up

PING 10.20.0.2 (10.20.0.2) 56(84) bytes of data.
From 10.30.0.1 icmp_seq=1 Destination Port Unreachable

PING 10.20.0.3 (10.20.0.3) 56(84) bytes of data.
From 10.30.0.1 icmp_seq=1 Destination Port Unreachable
...
```

只有10.20.0.1（网关）通，其余IP全部不可达，返回"Destination Port Unreachable"。

**失败原因分析：**

1. **防火墙默认策略拦截**：FORWARD链默认策略为DROP，所有未经明确允许的跨区域流量均被拦截。guest到office的流量没有匹配任何放行规则。

2. **REJECT规则生效**：规则11（LOG）和规则12（REJECT）明确拒绝了guest到office的所有流量。ICMP请求被拦截后返回"Destination Port Unreachable"。

3. **内网主机不可见**：攻击者只知道网关地址，无法探测到任何办公网主机（10.20.0.2-10.20.0.10），内网资产得到有效隐藏。

**防御有效性评估：**

该攻击被完全拦截。防火墙成功阻止了攻击者对内网结构的探测，仅暴露了网关地址。在真实攻击场景中，攻击者无法通过ICMP扫描获取内网资产信息，横向移动的第一步被阻断。

### 2.3 攻击2：尝试绕过防火墙访问dmz:22

**攻击目标：** 改变源端口，尝试绕过防火墙对guest→dmz:22的访问控制。

**攻击命令：**
```bash
sudo ip netns exec guest curl --local-port 80 --max-time 2 http://10.40.0.2:22/
sudo ip netns exec guest curl --local-port 443 --max-time 2 http://10.40.0.2:22/
```

**攻击结果：**

```
curl: (7) Failed to connect to 10.40.0.2 port 22 after 0 ms: Could not connect to server
curl: (7) Failed to connect to 10.40.0.2 port 22 after 0 ms: Could not connect to server
```

所有尝试均失败。

**失败原因分析：**

1. **五元组精细匹配**：防火墙规则基于完整的五元组（协议、源IP、源端口、目的IP、目的端口）进行匹配。guest到dmz:22的REJECT规则使用 `-i veth-fw-guest -o veth-fw-dmz` 匹配完整连接路径，不依赖源端口。

2. **规则优先级**：REJECT规则位置在FORWARD链中，匹配所有guest→dmz的流量。无论源端口如何变化，目的IP和目的端口固定为10.40.0.2:22，始终命中拒绝规则。

3. **状态检测补充**：即使尝试绕过，状态检测机制也会识别出这是非授权连接尝试。

**防御有效性评估：**

该攻击被完全拦截。防火墙基于连接特征而非单包特征进行访问控制，单纯改变源端口无法绕过规则。这体现了精细规则设计的有效性——拒绝规则应基于完整的连接信息，而非简单的端口匹配。

### 2.4 攻击3：尝试伪造VPN流量

**攻击目标：** 伪造源地址为10.10.10.2，冒充VPN用户访问内网资源。

**攻击命令：**
```bash
sudo ip netns exec guest ping -I 10.10.10.2 -c 1 10.20.0.2 2>&1
sudo ip netns exec guest curl --interface 10.10.10.2 --max-time 2 http://10.40.0.2:8080/ 2>&1
```

**攻击结果：**

```
ping: bind: 无法分配被请求的地址
curl: (45) bind failed with errno 99: 无法分配被请求的地址
```

**失败原因分析：**

1. **本地IP不存在**：10.10.10.2是VPN分配给remote的IP，不属于guest的本地接口。操作系统无法绑定不存在的IP地址，发送伪造源IP的包在本地就被拒绝。

2. **入口接口不匹配**：即使攻击者能构造伪造包，fw也会基于入口接口进行判断。guest的包从veth-fw-guest进入，而VPN流量从wg0进入。防火墙规则明确限制 `-i wg0 -s 10.10.10.2`，入口接口不匹配，规则不命中。

3. **反向路径过滤**：Linux内核的rp_filter会检查包的源IP是否可达，如果源IP不在该接口的合理路由路径上，包会被丢弃。

**防御有效性评估：**

该攻击从操作系统层面和防火墙层面均被拦截。入口接口匹配机制是防御源地址伪造的有效手段——即使攻击者伪造了VPN源IP，也无法从guest接口进入防火墙。这体现了"接口级信任"的安全设计思想：不同接口代表不同安全等级，信任不可跨接口传递。

### 2.5 REJECT与DROP的安全差异

本实验中两种拒绝方式的对比：

| 对比项 | REJECT | DROP |
|:-------|:-------|:-----|
| 响应行为 | 返回ICMP不可达 | 静默丢弃，无响应 |
| 攻击者感知 | 能确认目标存在 | 无法判断目标是否存在 |
| 信息暴露 | 暴露目标在线 | 不暴露任何信息 |
| 适用场景 | 内部违规访问 | 外部攻击防护 |
| 实验案例 | guest→office（REJECT） | internet→office（DROP） |

攻击者在扫描时，REJECT返回"Destination Port Unreachable"，可以判断目标主机在线但被拒绝；DROP则直接超时，无法判断是目标不存在还是被丢弃。因此，对外的安全策略应优先使用DROP。


## 三、防御方分析

### 3.1 从日志中识别攻击

**问题1：从日志的哪些字段可以判断这是来自guest的攻击？**

从 `IN=veth-fw-guest` 字段可以判断流量来自guest区域，`SRC=10.30.0.2` 确认攻击者IP，`OUT=veth-fw-office` 指示攻击目标区域，`DPT=22` 识别攻击的服务端口。结合 `PROTO=ICMP` 可以识别ping扫描。这些字段共同构建了攻击行为的完整画像，帮助分析人员快速定位攻击源、攻击目标和攻击手法。

**问题2：如果日志中`IN=veth-fw-guest OUT=veth-fw-office`，说明了什么？**

说明guest区域主机尝试访问office区域但被防火墙拦截。这是一次典型的跨区域违规访问，违反了"访客不能访问办公网"的安全策略。防火墙成功实施了区域隔离，阻止了未授权访问。在真实场景中，这种日志模式可能代表横向移动尝试、内网扫描或病毒传播，应立即关注并溯源分析。

**问题3：为什么看到大量相同来源的日志应该引起警惕？**

大量相同来源的拒绝日志表明该IP正在进行自动化扫描或暴力破解攻击，是入侵的前兆。这种模式通常标志着攻击者在进行信息收集或尝试突破访问控制。如果不及时干预，攻击者可能发现绕过方法或转向其他攻击路径。安全团队应立即对该IP进行封禁，并排查来源区域的安全状态。

### 3.2 规则计数器分析

**问题1：哪条规则拦截了guest访问office？**

规则12（行号12）：`REJECT all -- veth-fw-guest veth-fw-office`。该规则匹配所有从guest区域发出的、目标是office区域的流量，不分协议和端口。pkts=22，说明该规则已被触发22次。

**问题2：如果guest→office的规则计数很高，说明了什么？**

说明guest区域存在持续的违规访问尝试，可能有自动化工具在运行。这反映出三类风险：1）guest区域主机可能已被入侵，成为跳板机；2）访客网络存在恶意软件在进行横向移动；3）有人在主动进行渗透测试。应结合时间分布判断是人工操作还是自动化攻击，并采取相应措施。

**问题3：REJECT和DROP在安全性上有什么区别？**

REJECT返回ICMP不可达消息，攻击者能确认目标在线；DROP静默丢弃，攻击者无法判断目标是否存在，增加信息收集难度。从安全角度看，DROP更优。本实验中，内部guest→office使用REJECT便于运维，外部internet→office使用DROP保护资产。


## 四、边界测试改进

### 4.1 选择的问题

**dmz:8080对外开放（限制单IP连接数）**

DMZ的Web服务（端口8080）对外网完全开放，是攻击面的重要组成部分。在本次演练中，虽然攻击者无法直接访问未授权服务，但dmz:8080作为公开服务，面临以下风险：DDoS攻击耗尽服务资源、暴力破解Web登录接口、慢速连接长期占用资源、单IP建立数千并发连接消耗系统资源。这些风险可能导致服务不可用，属于高可用性威胁。

### 4.2 改进方案

限制单个源IP对dmz:8080的最大并发连接数为10。

**实现代码：**
```bash
sudo ip netns exec fw iptables -I FORWARD 1 \
  -p tcp --syn --dport 8080 \
  -d 10.40.0.2 \
  -m connlimit --connlimit-above 10 --connlimit-mask 32 \
  -j REJECT --reject-with tcp-reset
```

**规则解释：**

`-I FORWARD 1` 确保规则在最前面匹配；`-p tcp --syn` 只匹配新建连接，不干扰已建立的连接；`--connlimit-above 10` 设置并发连接数阈值；`--connlimit-mask 32` 精确到单个IP；`-j REJECT --reject-with tcp-reset` 快速释放资源，避免连接超时。

**测试效果：**
```bash
for i in {1..15}; do
  sudo ip netns exec internet curl -s -o /dev/null -w "连接$i: %{http_code}\n" --max-time 2 http://203.0.113.1:8080 &
done
wait
```

15个并发连接全部返回000，超过10个的连接被拒绝。规则有效防止了单IP的资源耗尽攻击。


## 五、演练总结

本次攻防演练验证了企业边界网络安全架构的有效性。攻击方的三类典型攻击（网段扫描、端口绕过、源地址伪造）均被防火墙成功拦截，防御方的日志分析和规则计数器能够准确识别攻击行为。边界测试发现并改进了dmz:8080的单IP连接数限制问题。

关键防御机制总结：

1. **区域隔离**：guest无法访问office和dmz，隔离机制有效
2. **精细规则**：基于五元组的访问控制，源端口变化无法绕过
3. **接口级信任**：VPN流量只能从wg0进入，源地址伪造被阻断
4. **REJECT vs DROP**：内部违规使用REJECT（便于排查），外部攻击使用DROP（不暴露信息）
5. **并发控制**：connlimit限制了单IP资源消耗，提升了服务可用性
