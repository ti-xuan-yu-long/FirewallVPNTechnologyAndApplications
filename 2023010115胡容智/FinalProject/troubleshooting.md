# 多区域边界靶场拓扑与 VPN 建立：典型故障排查报告

在整个实验方案的落地搭建与全链路打通调试中，遭遇了数次复杂的网络级故障。本报告严格依据三层网络路由与 Netfilter 状态机模型，记录并复盘四起核心故障的排查思路与攻坚方案。

---

## 故障 1：WireGuard 服务端与客户端完全无法握手，隧道持续断连

### 1.1 现象描述
执行 `wg-quick up wg0` 后，两端虚拟网卡接口均显示 `UP` 状态，但使用 `wg show` 查看时发现没有握手时间戳（`Latest handshake`），且两端互 ping 隧道 IP（192.168.100.1 与 192.168.100.2）完全不通。

### 1.2 排查路径（递进式诊断）
1.  **物理链路层检查**：在 `remote` 命名空间尝试直接 ping 防火墙公网口 `198.51.100.1`，结果反馈 `Network is unreachable`。
2.  **拓扑根因分析**：审查初始拓扑发现，`remote` 作为远程端，其网络接口与防火墙公网口分属于不同的物理孤立网段，由于缺少中间商（外部 Internet 网关），两端无法建立最基础的三层公网通信。
3.  **阻断点定位**：底层的物理/虚拟对等体（Endpoint）不通，上层的 WireGuard UDP 51820 握手报文根本无法送达对端。

### 1.3 解决方案
利用 Linux 内核网桥技术重构虚拟 Internet 二层广播域：
*   在 `external` 空间中创建虚拟桥接网卡 `br0`：
    ```bash
    ip netns exec external ip link add br0 type bridge && ip netns exec external ip link set br0 up
    ```
*   将 `fw` 外部公网口与 `remote` 物理接口通过两对全新的 `veth` 网卡对统一接入 `br0` 网桥。
*   为 `remote` 赋予静态公网 IP `198.51.100.100/24`。至此，两端物理对等点具备同网段点对点通信能力，WireGuard 握手瞬间成功拉起。

## 故障 2：WireGuard 隧道协商成功，但内网流量“有去无回”（回程断开）

### 2.1 现象描述
`wg show` 确认握手已经成功建立，从 `remote` 端可以顺利 ping 通防火墙隧道网关 `192.168.100.1`。但是当 `remote` 尝试通过隧道访问内网核心服务器 `172.16.30.2` 时，流量发生严重中断。

### 2.2 排查路径
1.  **去程路由审查**：在 `remote` 侧查验路由表（`ip route`），存在 `172.16.30.0/24 dev wg0` 路由，说明去程流量已被成功引入加密隧道。
2.  **网关转发状态**：在防火墙网关（`fw`）抓包，发现去程报文已解密并成功转发给了 `server`，并且 `server` 已经发回了回程响应报文。
3.  **密钥路由冲突机制**：问题锁定在 `fw` 的 WireGuard 驱动层。审查 `/etc/wireguard/fw/wg0.conf`，发现 `[Peer]` 字段下的 `AllowedIPs` 误配置为了 `0.0.0.0/0`。

### 2.3 解决方案
根据 WireGuard 的 `Cryptokey Routing`（加密密钥路由）机制，当回程流量（源：172.16.30.2，目的：192.168.100.2）准备塞回隧道时，网关内核驱动会强行校验目的 IP 是否匹配 `AllowedIPs`。
*   将 `fw` 端的客户端 Peer 规则进行重构：
    ```ini
    [Peer]
    PublicKey = <REMOTE_PUBLIC_KEY>
    AllowedIPs = 192.168.100.2/32
    ```
*   这强行促使网关在内核路由表中生成了一条精确指向隧道内特定主机的明细回程路由，完美解决了流量非对称丢失故障。

## 故障 3：DNAT 地址转换映射成功，但外网客户端依然无法访问

### 3.1 现象描述
在防火墙的 `nat` 表中灌入了标准的 `PREROUTING` 规则，将公网 `8080` 映射至内网 `3000`。然而外网用户通过浏览器或 `curl` 请求 `198.51.100.1:8080` 时，始终卡死在连接握手阶段，最终返回超时失败。

### 3.2 排查路径
1.  **NAT 表项跟踪**：执行 `iptables -t nat -L -v -n`，发现 `PREROUTING` 链上的命中计数器正在规律增长，证明目的地址转换（DNAT）已经生效，报文头部已被改写。
2.  **网络层路由流向**：经过 DNAT 头改写后，数据包的物理归属从流向防火墙自身的 `INPUT` 链，瞬间变更为了跨安全域转发的 `FORWARD` 链。
3.  **安全基线冲突**：由于我们在第七部分为了推行零信任网络，统一将 `FORWARD` 链的默认动作置为了 `DROP`（`iptables -P FORWARD DROP`），而当时并未在 `FORWARD` 链中显式为转换后的内网真实 IP 配置通行证。

### 3.3 解决方案
在防火墙的安全过滤策略脚本（`firewall.sh`）中，前置补充一条针对转换后真实内网宿主机和业务端口的 `FORWARD` 链放行规则，建立“转换 + 放行”的联动闭环：

sudo ip netns exec fw iptables -I FORWARD 1 -d 172.16.30.2 -p tcp --dport 3000 -j ACCEPT
